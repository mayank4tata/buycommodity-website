# BCSPL Website V2.1.4 Deployment Guide

## Deployment order

Complete the backend steps before replacing the live website files.

### 1. Back up the live database and website

Export the current product masters, pricing and grade master from the Admin Panel. Keep the current V2.1.3 ZIP and a database backup before proceeding.

### 2. Run the database migration

Open Supabase **SQL Editor** and run:

`supabase/migrations/20260724_v2_1_4_admin_access.sql`

The migration:

- adds `full_name`, `role`, `active`, invitation and timestamp fields to `admin_users`;
- defines `public.is_admin()` and `public.is_super_admin()`;
- adds the `admin_access_audit` table;
- updates RLS so users can read their own admin profile and Super Admins can read all profiles;
- automatically promotes the earliest active existing administrator to `super_admin` only when no Super Admin exists.

Verify the selected Super Admin before continuing:

```sql
select user_id, email, full_name, role, active
from public.admin_users
order by created_at;
```

To change the initial Super Admin manually:

```sql
update public.admin_users
set role = 'super_admin', active = true
where lower(email) = lower('YOUR-ADMIN-EMAIL');
```

### 3. Configure Supabase Auth URLs

In **Supabase Dashboard → Authentication → URL Configuration** set:

- Site URL: `https://www.buycommodity.in`
- Additional Redirect URL: `https://www.buycommodity.in/admin-reset-password.html`
- Optional non-www redirect: `https://buycommodity.in/admin-reset-password.html`

Keep localhost redirects only during testing.

### 4. Deploy the user-management Edge Function

Function folder:

`supabase/functions/manage-admin-users`

Using the Supabase CLI from the project folder:

```bash
supabase functions deploy manage-admin-users
```

Keep JWT verification enabled for this function. It is an authenticated Super Admin function. Do not expose the service-role key in any website file.

Optional function secret:

```bash
supabase secrets set ADMIN_SITE_URL=https://www.buycommodity.in
```

`SUPABASE_URL`, `SUPABASE_ANON_KEY` and `SUPABASE_SERVICE_ROLE_KEY` are normally available automatically inside Supabase Edge Functions.

### 5. Upload the website files

Replace the live source with all files from `01_Source Code`. The following new files must be included:

- `admin-reset-password.html`
- `css/v2.1.4-admin-access.css`
- `js/admin-reset-password.js`

Also deploy the revised:

- `admin.html`
- `js/app.js`

### 6. Test before general use

1. Sign in with the existing Super Admin.
2. Confirm **Users & Access** appears in the sidebar.
3. Invite a test Admin user.
4. Open the invitation email and set a password.
5. Sign in as the test Admin and confirm **Users & Access** is not visible.
6. Use **Forgot password?** from the login screen and complete the reset.
7. Disable the test user and confirm their next admin request is rejected.
8. Reactivate the user and test login again.
9. Check `admin_access_audit` for recorded access events.

### Rollback

The public catalogue is not dependent on the new user-management function. For a front-end rollback, restore V2.1.3 website files. The added database columns and audit table may safely remain. Do not remove `public.is_admin()` because the existing catalogue RLS depends on it.
